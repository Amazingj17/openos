## 6. 阻塞缺陷 R1：缺少逐 seed mean

ADR 第 3.4 节明确要求“至少 5 个训练 seed，公开全部 seed-level mean”。当前 `_aggregate_policy` 已计算 `seed_means`，但输出只保留：

```text
seed_count
seed_mean_ratio_std
```

本次两个主策略 seed 的 mean 都是 `1.0`，报告中却没有 seed `101→1.0`、`202→1.0` 的映射。只有总体标准差无法判断离群 seed，也不能证明 5-seed 行来自五次独立运行。

B 修复时应冻结稳定 schema，至少在 JSON 中输出有序的：

```json
"seed_mean_ratios": [
  {"seed": 101, "mean_ratio": 1.0},
  {"seed": 202, "mean_ratio": 1.0}
]
```

推荐同时新增 `evaluation_per_seed.csv` 并纳入 manifest，便于论文表格和人工审计。测试必须覆盖不同 seed mean，不能只用本轮两个均值都为 1 的例子。

## 7. 阻塞缺陷 R2：`receipt_name` 未执行

独立契约冻结：

```text
receipt_name = public_test_gate_receipt.json
```

A 使用同一份 contract hash、release commit、A/B 列表和 nonce，实际成功领取：

```text
receipt-one.json
receipt-two.json
```

随后 final report 传入 `receipt-one.json` 仍被接受。也就是说，当前“同一路径二次拒绝”正确，但契约中的唯一文件名没有参与 claim 或 final receipt 校验；更换路径即可重复领取。

B 至少必须：

1. claim 前验证 `Path(receipt).name == contract.test_gate.receipt_name`；
2. receipt payload 写入冻结文件名；
3. final report 再验证传入路径 basename 和 receipt 字段；
4. 增加错误文件名、同授权不同文件名、final 错误文件名三个拒绝测试。

这仍不能形成全局不可删除账本；正式 loader 固定 release artifact 目录和非密码学边界继续保留为后续任务，不能因此宣称安全问题已彻底解决。

## 8. 阻塞缺陷 R3：伪 UTC 时间被接受

A 将授权字段设置为：

```json
"authorized_at_utc": "not-a-timestamp"
```

claim 成功，receipt 被创建。仅检查“非空字符串”不能支撑时间顺序审计。

B 应在契约中冻结并实现严格 UTC 格式，例如 `YYYY-MM-DDTHH:MM:SSZ`；解析后还应按同一 canonical 格式写 receipt。至少拒绝任意文本、缺时区、无效日期和非 UTC offset，并保留正确 leap-day/边界测试。
