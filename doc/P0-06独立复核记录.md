## 6. 校验值与平台换行

实现文档中的源码 SHA-256 是按 Git canonical LF 字节计算的。Windows 全局 `core.autocrlf=true` 会让 fresh worktree 使用 CRLF，直接对工作树字节计算会得到不同值；把 CRLF 规范化为 LF 后，四个文件分别与接口文档中的 `d1e40b...`、`74d980...`、`e18f36...`、`03c5d7...` 完全一致。这不是源码差异，但发布 manifest 必须明确 hash 的字节规范。

## 7. 限制与结论

- 当前 evaluator 对 scheduler 失败仍是 fail-fast，成功批次的 `valid_schedule_rate=1.0` 不能替代真实失败率；P0-08 必须保留失败实例并输出 failure rate。
- adapter 执行团队已审计命令，但不提供容器、网络、用户或文件系统沙箱。
- 每场景进程启动开销计入 runtime；正式算法性能报告必须区分 adapter 开销。
- 本次只检查直接超时子进程已清理，不声称能终止外部程序自行创建的任意后代进程树。
