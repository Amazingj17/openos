## 1. 固定计算规则

任务 `i` 在资源 `r` 上的执行时间：

\[
w_{i,r}=workload_i/speed_r
\]

平均执行时间：

\[
\bar w_i=\frac{1}{|R|}\sum_{r\in R}w_{i,r}
\]

跨资源通信时间：

\[
c_{i,j}^{r_s,r_t}=latency_{r_s,r_t}+data_{i,j}/bandwidth_{r_s,r_t}
\]

同资源通信为 0。HEFT rank 使用不同资源有序对上的平均通信成本；只有一个资源时平均通信为 0。

\[
rank_u(i)=\bar w_i+\max_{j\in succ(i)}(\bar c_{i,j}+rank_u(j))
\]

调度规则固定为：

1. 只在 ready tasks 中选择 upward rank 最大者；rank 相同选 task ID 小者。
2. 对选中任务计算每个资源的插入式 earliest finish time，选择完成最早的资源；完成时刻相同选 resource ID 小者。
3. 新任务可插入资源时间线已有区间之前或之间，不要求按决策顺序追加。
4. 所有期望值先人工推导，再写入只读 fixture；测试代码不得在运行时重新生成期望值。

## 3. 逐例推导

### G01：链式 DAG，同资源避免通信

- workload：`T0=4, T1=2`；`T0→T1` 数据量 2。
- 资源速度：`R0=1, R1=2`；跨资源带宽 2、时延 0。
- 平均执行：`T0=3, T1=1.5`；平均通信为 1。

```text
rank(T1) = 1.5
rank(T0) = 3 + 1 + 1.5 = 5.5
```

`T0` 在 R1 为 `0–2`。`T1` 留在 R1 为 `2–3`；若迁移到 R0，需要通信 1，再执行 2，于 `5` 完成。因此 makespan 为 `3`。

### G02：fork 与确定性 tie-break

- workload：`T0=2, T1=4, T2=4`；`T0→T1/T2`，通信为 0。
- 资源速度：`R0=1, R1=2`。

```text
rank(T1) = rank(T2) = mean(4, 2) = 3
rank(T0) = mean(2, 1) + 3 = 4.5
```

`T0` 先在 R1 执行 `0–1`。T1/T2 rank 相同，先选 ID 小的 T1，在 R1 执行 `1–3`。T2 在 R0 和 R1 都于 5 完成，选择 ID 小的 R0，区间为 `1–5`。makespan 为 `5`。

### G03：diamond 与 HEFT 局部短视

- workload：`T0=4, T1=4, T2=8, T3=2`。
- 资源：`R0 speed=2`，`R1 speed=4`。
- 边：`T0→T1:2`、`T0→T2:4`、`T1→T3:2`、`T2→T3:2`。
- `R0→R1` 为 `0.5+data/2`；`R1→R0` 为 `0.25+data/4`。

数据量 2 的平均通信为 `(1.5+0.75)/2=1.125`；数据量 4 为 `(2.5+1.25)/2=1.875`。

```text
rank(T3) = 0.75
rank(T1) = 1.5 + 1.125 + 0.75 = 3.375
rank(T2) = 3 + 1.125 + 0.75 = 4.875
rank(T0) = 1.5 + max(1.125+3.375, 1.875+4.875) = 8.25
```

HEFT 调度：`T0@R1 0–1`、`T2@R1 1–3`、`T1@R0 1.75–3.75`、`T3@R0 3.75–4.75`，makespan `4.75`。

合法替代方案全部放在 R1：`T0 0–1`、`T2 1–3`、`T1 3–4`、`T3 4–4.5`。其 ratio 为 `4.5/4.75=0.9473684211`，证明 HEFT 为 T1 选择局部更早的 R0 会损害汇合任务。

### G04：单资源链

- workload：`[3,2,5]`，资源 R0 速度 2。
- 两条边数据量均为 100，但同资源通信强制为 0。
- 执行时间：`[1.5,1,2.5]`。

```text
rank(T2)=2.5
rank(T1)=1+2.5=3.5
rank(T0)=1.5+3.5=5.0
```

唯一调度为 `T0 0–1.5`、`T1 1.5–2.5`、`T2 2.5–5`，makespan `5`。该案例防止实现错误地向同资源依赖收取通信费用。

### G05：后决策任务插入早期空隙

- workload：`T0=40, T1=1, T2=20, T3=10`；只有 `T1→T2`，数据量 10。
- 资源速度：`R0=1, R1=10`；跨资源带宽 1，通信时间 10。
- 平均执行分别为 `22, 0.55, 11, 5.5`。

```text
rank(T0)=22
rank(T2)=11
rank(T1)=0.55+10+11=21.55
rank(T3)=5.5
```

决策与时间线：

1. T0 在 R1 执行 `0–4`。
2. T1 在 R0 执行 `0–1`。
3. T2 若留 R0 于 21 完成；迁移 R1 需等通信到 11，再执行 `11–13`，故选择 R1。
4. T3 最后决策，但 R1 的 `4–11` 有空隙，插入为 `4–5`，而不是追加到 13 之后。

makespan 为 `13`。该案例直接验证插入式 earliest-slot 实现。

### G06/G07：只交换链路方向，调度发生改变

两个案例具有相同任务和资源：

- 两个根任务 `T0(work=8)`、`T1(work=4)` 汇合到 `T2(work=2)`，边数据量均为 2。
- `R0 speed=1`，`R1 speed=2`。
- 平均执行为 `6,3,1.5`；两个方向通信分别为 2 和 0.5，平均 1.25。
- ranks 均为 `[8.75,5.75,1.5]`。
- 前两步均为 `T0@R1 0–4`、`T1@R0 0–4`。

G06 中 `R0→R1=2`、`R1→R0=0.5`：

```text
T2@R0: start=max(4, 4+0.5)=4.5, finish=6.5
T2@R1: start=max(4, 4+2)=6, finish=7
```

所以选择 R0，makespan `6.5`。

G07 交换方向，`R0→R1=0.5`、`R1→R0=2`：

```text
T2@R0: start=6, finish=8
T2@R1: start=4.5, finish=5.5
```

所以选择 R1，makespan `5.5`。这对案例禁止把非对称通信矩阵误当作对称矩阵。

### G08：相同任务与相同资源的 tie-break

两个独立任务 workload 均为 2，三个资源速度均为 1，因此两任务 rank 均为 2。

- 先选 task ID 0；三个资源均在 2 完成，选 resource ID 0。
- 再选 task ID 1；R1/R2 均可在 2 完成，选 resource ID 1。

调度为 `T0@R0 0–2`、`T1@R1 0–2`，makespan `2`。该案例保证跨平台结果确定。

### G09：数据量为零不等于通信时间为零

任务结构和资源速度与 G06/G07 相同，但两条边数据量为 0，跨资源固定时延为 1。

平均通信仍为 1，因此 ranks 为 `[8.5,5.5,1.5]`。前两任务分别在 R1、R0 执行 `0–4`。T2 在 R1 需等待 T1 的固定时延到 5，再执行 `5–6`；在 R0 则完成于 7，所以选择 R1，makespan `6`。

该案例防止实现以 `data=0` 为由跳过非零 latency。

### G10：断连子图与资源可用性

- `T0(work=4)→T2(work=2)`；T1(work=6) 与该链断连。
- 资源速度 `R0=1, R1=2`，通信为 0。

```text
rank(T2)=1.5
rank(T0)=3+1.5=4.5
rank(T1)=4.5
```

T0/T1 rank 相同，先选 T0，在 R1 执行 `0–2`。T1 随后仍选择 R1，执行 `2–5`。T2 此时在 R0 可于 4 完成，在 R1 要到 6，故选择 R0 执行 `2–4`。makespan 为 `5`。
