# P1-06 最终模型与结果包双签规程

- 任务：`P1-06`
- 主责/复核：成员 A 与成员 B 互审
- 状态：构建与验收工具已完成；正式执行受 P1-A05、独立复核和 G3 阻塞
- 机器合同：[`configs/p1_06_bundle_contract.json`](../configs/p1_06_bundle_contract.json)

## 1. 进入条件

只有以下条件全部满足才创建 G3 授权和 P1-06 review request：

1. 唯一 P1-A05 正式训练成功，五 seed 均无 failure/illegal，训练 summary、manifest 和 checkpoint hash 可重算。
2. 唯一四切片 Development evidence 与标准报告已经生成。
3. `comparison.json` 的六个 development gate 全为 `true`，决定为 `eligible_for_independent_review_before_G3`。
4. B 已从不可变候选提交独立复核训练 artifact、逐 checkpoint validation、evidence、report 和 comparison。
5. A/B 在 G3 都明确批准 P1-A05 为主模型，release 工作树干净。

若 comparison 为负结果，立即保留 P1-A04、固化负结果并停止 P1-A05；不得创建伪造的 G3 授权或 P1-06 回执，也不得访问 public test。

## 2. 最小消融与包内容

P1-A05 只有一个预注册干预：把训练 rollout 从冻结的 P1-A04 方案改为等 transition 的 STG-50/synthetic-100 混合，其余 seed、模型、优化器、epoch、奖励、validation 选模和 Development 场景不变。因此，P1-A05 与 P1-A04 的同 training-seed×同 scenario 配对 comparison 就是 P1-06 的最小消融，不再训练第二个候选。

最终模型/结果 zip 只包含：

- 五个主 actor checkpoint；
- P1-A05 summary、run manifest、配置/预注册/实现复核 provenance；
- P1-A04 baseline 与 P1-A05 candidate 的完整 Development evidence；
- 标准报告的 JSON、manifest 和三份 CSV；
- P1-A05−P1-A04 配对 comparison；
- B 的 P1-A05 独立 replay receipt、G3 双人授权、P1-06 review request 和 A/B 两份独立回执。

内部 `MODEL_RESULT_MANIFEST.json` 绑定每个文件的来源相对路径、用途、字节数和 SHA-256。zip 顺序、时间和权限固定；同一输入连续构建必须逐字节一致。包中不包含训练 state、raw benchmark、public-test 字节、授权凭据、私钥或源码工作树。

## 3. G3 授权格式

G3 通过后，在干净 release commit 上由 A/B 共同创建被 Git 忽略的 `outputs/p1-06/g3_authorization.json`：

```json
{
  "format_version": 1,
  "task": "G3",
  "candidate_commit": "<comparison.code.commit 的 40 位提交>",
  "release_commit": "<git rev-parse HEAD 的 40 位提交>",
  "comparison_sha256": "<comparison.json 文件 SHA-256>",
  "independent_review_sha256": "<p1_a05_independent_review.json 文件 SHA-256>",
  "development_gate_passed": true,
  "decision": "approve_p1_a05_as_primary",
  "test_accessed": false,
  "public_test": "forbidden",
  "approvals": [
    {"member": "A", "decision": "approve"},
    {"member": "B", "decision": "approve"}
  ]
}
```

构建器要求成员集合恰好为 `A/B`、两人均批准、candidate/release commit 与 comparison/hash 完全一致，并且 G3 绑定 B 在不可变远端候选提交上生成的独立 replay receipt。该 receipt 的生成与篡改拒绝规则见[P1-A05 正式运行与 G3 收口规程](./P1-A05正式运行与G3收口规程.md)。

## 4. 生成待复核清单

```powershell
git status --porcelain --untracked-files=no
python scripts/build_p1_06_bundle.py prepare-review
python -c "import json,pathlib; from trisched.hashing import canonical_json_sha256; p=pathlib.Path('outputs/p1-06/p1_06_review_request.json'); print(canonical_json_sha256(json.loads(p.read_text(encoding='utf-8'))))"
```

`prepare-review` 只读现有 artifact，验证 comparison、B 独立 replay receipt、G3、五 checkpoint、报告清单和 public-test 边界，再一次性写出 `p1_06_review_request.json`；已有文件时拒绝覆盖。该 request 的 `package_artifacts` 是 A/B 要逐项重算的冻结清单。

## 5. A/B 双签格式

A 与 B 不共享审计结果，各自从 request 的来源路径重算 hash、核对主表/最小消融、随机复评 checkpoint，并分别创建 `p1_06_review_a.json` 和 `p1_06_review_b.json`：

```json
{
  "format_version": 1,
  "task": "P1-06",
  "member": "A",
  "decision": "approve",
  "request_canonical_sha256": "<review request 的 canonical SHA-256>",
  "candidate_commit": "<request.candidate_commit>",
  "release_commit": "<request.release_commit>",
  "public_test_accessed": false,
  "assertions": {
    "checkpoint_hashes_recomputed": true,
    "development_result_hashes_recomputed": true,
    "minimal_ablation_reviewed": true,
    "package_inventory_reviewed": true
  }
}
```

B 的文件只把 `member` 改为 `B`。任何断言不能确认时必须写负结论并停止，不能把字段填成 `true` 来通过工具。构建器拒绝单人回执、重复成员、错误 commit/request hash、漏项、artifact 漂移或 public-test 访问。

## 6. 构建、复验与提交规范

```powershell
python scripts/build_p1_06_bundle.py build
python scripts/build_p1_06_bundle.py verify `
  outputs/release/trisched-model-results-<release_commit12>.zip

python scripts/build_p1_06_bundle.py build `
  --output outputs/release/trisched-model-results-<release_commit12>-repeat.zip
Get-FileHash outputs/release/trisched-model-results-<release_commit12>*.zip -Algorithm SHA256
```

两份 zip 的 SHA-256 必须相同。提交结果时记录：

- `candidate_commit`、`release_commit` 和 source zip SHA-256；
- model/result zip 文件名、字节数、SHA-256 和离线备份位置；
- 五 checkpoint 的 seed、文件名、文件 SHA-256、参数 SHA-256；
- 四切片 mean ratio、size 配对 CI、failure/illegal 数和 comparison 决定；
- B independent-review、G3 authorization、P1-06 A/B receipt 的文件 SHA-256；
- 完整验证命令、退出码、运行环境和已知限制。

模型/结果二进制保留在 `outputs/` 或正式制品库，不提交源码 Git。Git 只提交构建器、合同、测试、规程和不含二进制结果的审计记录。

## 7. Public-test 边界

P1-06 完成仍不授权 public test。最终测试需要新的 A/B 明确授权、冻结 release commit、loader 前一次性 receipt 和原子门禁；本工具不创建 test receipt、不调用 `claim-test-gate`，也不以 Development 报告中预期的 `release_publishable=false` 为理由提前访问 test。
