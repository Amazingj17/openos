## 1. 一页结论：怎样把作品做成获奖项目

> **总策略：**把作品定义成“可插拔的 DAG 调度实验平台”，而不是一个只能运行单一模型的训练脚本。所有创新都必须同时改善验证集指标、泛化或工程效率中的至少一项。

赛题的得分结构非常清晰：功能完整性 40 分、性能优化 35 分、文档质量 25 分。最优投入顺序应是“正确性与接口 → HEFT 可比基线 → 学习策略 → 泛化与统计 → 文档和演示”。若前两项不牢，模型再复杂也无法形成可信证据。

### 1.1 获奖的三条硬标准

1. **可信：**环境转移、通信代价、资源占用和 makespan 重算必须经过独立验证；结果不能只相信 reward。
1. **领先：**在固定隐藏测试集上，RL/HEFT 的 mean\_ratio 小于 1，且优势跨随机种子、图规模和异构程度保持。
1. **可复现：**评委只需一条命令即可完成训练或加载模型、批量评测并生成标准结果文件。

### 1.2 产品定义与范围

作品名暂定为 TriSched。它接收任务 DAG、云/边/端节点配置与通信矩阵，通过统一 Scheduler 接口生成合法调度，输出 makespan、相对 HEFT 比值、推理时间和完整轨迹。框架支持启发式、行为克隆和 DRL 策略的无差别批量评测。

两人团队默认完整交付 P0，并在门禁通过后集中完成一条 P1 主路线。P2 不纳入默认排期；只有主结果、复现和文档均已冻结，且剩余工期不少于两周时，才允许选择一个 P2 亮点验证。

## 2. 赛题建模与验收口径

给定有向无环图 G=(V,E) 与异构资源集合 R。任务 v 在资源 r 上的执行时间由工作量与资源速度共同决定；若前驱和后继位于不同资源，还需计入数据量/链路带宽形成的通信时间。调度器每一步从就绪任务中选择一个任务，并分配到合法资源，目标是最小化所有任务完成时刻的最大值 makespan。

### 2.1 状态、动作、约束和奖励

***表 3  问题建模与代码对象的一致性约定***

| **要素** | **建议定义** | **实现验收** |
| --- | --- | --- |
| 任务图 | 节点：工作量、资源执行时长、入/出度、上行 rank、关键路径余量、完成状态；边：数据量与依赖 | 拓扑排序稳定；循环输入拒绝；特征归一化参数固化到 checkpoint |
| 资源图 | 节点：类型、速度、可用时刻、负载；边：带宽、时延、可达性 | 支持 3–16 节点与非对称带宽；不可达边进入资源 mask |
| 动作 | 优先采用两阶段因子化动作：先选 ready task，再选 resource；联合概率保持可追溯 | task mask 与 resource mask 都在采样和贪心推理前作用于 logits |
| 转移 | 按插入式 earliest-start/finish 规则更新资源时间轴和通信就绪时刻 | 独立 makespan 重算器与环境累计值逐实例一致 |
| 奖励 | 每步负增量 makespan/下界 shaping + 终局 −makespan；可加 HEFT 相对改进项 | 无非法调度可获得高 reward；训练目标与 mean\_ratio 单调相关 |

### 2.2 关键指标

- **主指标：**mean\_ratio = mean(RL\_makespan / HEFT\_makespan)。越低越好；< 1 表示平均优于 HEFT。
- **可信度：**至少 5 个训练随机种子；测试集固定且不参与调参；报告样本数、均值、标准差和 bootstrap 95% CI。
- **补充指标：**win/tie/loss、P50/P95 ratio、SLR、相对最优 gap（小图 CP-SAT）、推理时延、训练时长和峰值内存。
- **泛化指标：**图规模、DAG 家族、CCR、资源数量、速度异构、带宽波动分别做 OOD 切片，禁止只报总体均值。

### 2.3 数据划分防泄漏规则

1. 先生成或导入场景，再以 topology\_hash + resource\_hash 去重。
1. 按随机种子和 DAG 家族划分 train/validation/test；test 清单提交前冻结并只由评测命令读取。
1. 归一化统计仅用 train；超参只看 validation；最终结果只跑一次 test 汇总。
1. 每个结果写入 config\_hash、dataset\_hash、checkpoint\_hash、git\_commit 与依赖版本。

## 3. 系统架构与统一接口

> **架构原则：**环境、场景、策略、训练和评测五层解耦。任何新算法只能依赖稳定接口，不得读取环境内部私有状态；否则无法证明公平比较。

***表 4  TriSched 分层架构***

| **层** | **核心职责** | **主要组件** | **验收证据** |
| --- | --- | --- | --- |
| 场景层 | 生成、加载、校验、划分 DAG 与资源参数 | ScenarioSchema、STG/Pegasus loader、SyntheticGenerator、SplitRegistry | 同一 seed 字节级复现；错误输入给出明确诊断 |
| 环境层 | 维护 ready set、通信、资源时间线、合法动作和终局指标 | HeteroDagEnv、MaskBuilder、ScheduleValidator、MakespanOracle | 黄金案例、属性测试、双实现交叉校验 |
| 策略层 | 统一启发式/BC/RL/外部策略调用 | Scheduler Protocol、HEFT、CPOP、Random、BC、MaskedPPO | 评测器不含策略分支；新增策略无需改环境 |
| 训练层 | 采样、课程、更新、早停、checkpoint | Trainer、Replay/Rollout、Curriculum、SeedManager | 断点续训一致；训练日志可关联配置 |
| 评测层 | 批量运行、统计、图表、报告 | Evaluator、SummaryWriter、ReportGenerator | summary.json + per\_instance.csv + 图表一键生成 |

### 3.1 建议仓库结构

```text
openos/
├─ configs/                 # 场景、算法、训练、评测配置
├─ src/trisched/
│  ├─ core/                 # 类型、schema、随机种子、日志
│  ├─ scenarios/            # 生成器、STG/Pegasus loader、split
│  ├─ envs/                 # 环境、mask、转移、独立 validator
│  ├─ policies/             # HEFT/CPOP/Random/BC/PPO/外部适配器
│  ├─ models/               # task/resource GNN、actor、critic
│  ├─ training/             # rollout、课程、早停、checkpoint
│  ├─ evaluation/           # 指标、统计、报告
│  └─ cli.py
├─ tests/                   # unit / property / integration / golden
├─ scripts/                 # 一键训练、评测、复现
├─ docs/                    # 设计、接口、复现、实验、限制
└─ outputs/<run_id>/         # config、模型、日志、结果与图表
```

### 3.2 策略接口最小契约

```text
reset(scenario, seed) -> None
act(observation, task_mask, resource_mask, deterministic) -> (task_id, resource_id)
observe(transition) -> None              # 启发式可为空实现
save(path) / load(path) -> None
metadata() -> {name, version, deterministic, training_info}
```

外部策略通过 adapter 接入相同契约。评测器只认该接口和标准 observation，不直接 import 某个算法类，由此满足“至少一种外部调度方法可插拔”的评分要求。

### 3.3 一键命令与标准产物

```text
python -m trisched.cli pipeline --config configs/competition.yaml
# smoke → train/load → validation/test → summary.json → report.md
```

- summary.json：主指标、切片指标、均值/方差/CI、训练和推理开销。
- per\_instance.csv：scenario\_id、HEFT/RL makespan、ratio、合法性、推理耗时。
- manifest.json：数据、配置、模型、代码、系统和依赖版本哈希。
- report.md：自动汇总表、关键图、失败案例和限制；技术报告引用该文件而不手抄数字。

### 4.1 推荐主模型：Constraint-Aware Dual-Graph PPO

1. **任务编码器：**对 DAG 做有向消息传递/Graph Attention，聚合前驱完成、后继压力、关键路径与通信特征。
1. **资源编码器：**对云/边/端节点及链路编码，显式表达速度、可用时刻、带宽、时延和可达性。
1. **任务策略头：**只对 ready tasks 输出 logits，并在 softmax 前执行严格 task mask。
1. **资源策略头：**以选中任务为条件，对合法资源打分；通信代价和 earliest finish 作为显式特征。
1. **价值网络：**使用图级池化和剩余工作下界，降低不同规模场景的 value 方差。
1. **教师/安全层：**先模仿 HEFT，再用 PPO 优化；推理低置信度时回退 HEFT 或在 HEFT 解附近做局部改进。

### 4.2 奖励与训练策略

- 奖励优先使用归一化的 Δmakespan 或剩余下界变化；终局以 −makespan/HEFT\_makespan 对齐主指标。
- 先小图、低异构、低 CCR，再逐步增加任务数、资源数和通信压力；课程推进由验证成功率触发。
- 每批混合 DAG 家族和资源配置，做速度/带宽扰动；禁止一个固定规模训练到底。
- 固定 HEFT teacher 数据集做 BC warm start；PPO fine-tune 使用 clipped objective、GAE、梯度裁剪和早停。
- 保存 best-by-validation-ratio 与 last 两类 checkpoint；所有比较使用相同场景顺序与随机种子。

### 5.2 公平比较规则

- 所有策略接收完全相同的场景对象、通信模型、插入式时间线和 wall-clock 限制。
- HEFT 既是基线又是 teacher，但 test 上不能用 RL 见到的标签调参；回退触发规则必须在 validation 冻结。
- 统计时逐场景配对；ratio 先逐实例计算再求均值，不能用两个总体均值相除。
- 失败、超时和非法结果不得丢弃；按预先定义的惩罚值计入，并单列 failure rate。
- 论文中的百分比仅作为研究参考，比赛结果只使用本项目统一环境重新测得的数据。

当前可执行的最小契约已落在 [P1-B02 评测契约与自动报告](./P1-B02评测契约与自动报告.md)：主策略与 Random 固定 5 seeds；ID 为 30 个冻结 validation 场景；OOD-Size 先固定 30 个 100-task 场景，另加 CCR/System 两个确定性变换切片；失败惩罚为 `10.0`；报告使用 10,000 次 seed 与共享场景交叉重采样。B 生成的[正式 Development 候选](./P1-B02正式Development结果记录.md)已由 A [独立复跑并复核通过](./P1-B02正式Development独立复核记录.md)：ID 和 CCR-OOD 稳健优于 HEFT，system-OOD 不确定，size-OOD 退化到 `1.568302`，五个 seed 均未过 1.0。原始失败结果和 `release_publishable=false` 必须保留；表 7 的更大 150/300/500-task 和 8–16-resource 矩阵仍是后续扩展目标，不能冒充当前已有证据。

| 工程效率 | 不同图规模/资源数 | 评测脚本是否可批量使用？ | 训练时间、推理 P95、内存 |

### 9.1 明确不做

- 不在环境和 HEFT 未验证前训练大模型。
- 不以训练 reward 代替调度合法性和独立 makespan。
- 不在 test 上选 checkpoint、调奖励或调回退阈值。
- 不把 UI、集群化和动态故障模拟放在一键复现之前。
- 不宣称“优于现有方法”而不提供固定场景、配对统计和可重跑脚本。
