## 1. 目标与边界

P0-06 允许不修改 TriSched 核心代码即可接入外部调度器。内置策略、学习策略和外部进程统一实现“输入一个 Scenario、返回一份完整 ScheduleResult”的 runner 契约，再进入同一 evaluator、生产 validator 和独立 validator。

外部协议是一场景一次子进程的 JSON-over-stdio 协议，不是逐动作的在线 RPC。它适合接入其他语言、独立 Python 环境或论文基线；不适合需要在每个环境 step 与训练器高频交互的在线 RL。

## 2. 配置 registry

`evaluation.schedulers` 是有序数组。字符串从内置 registry 创建 runner，对象创建外部进程 adapter。统一 evaluator 必须包含唯一的 `heft` 和 `masked_mlp`，因为当前主指标以 HEFT 为分母、以 masked MLP 为主策略。

默认配置：

```json
{
  "evaluation": {
    "random_seed": 991,
    "schedulers": ["heft", "cpop", "greedy_eft", "random", "masked_mlp"]
  }
}
```

接入仓库内参考 external HEFT：

```json
{
  "evaluation": {
    "random_seed": 991,
    "schedulers": [
      "heft",
      "cpop",
      "greedy_eft",
      "random",
      "masked_mlp",
      {
        "type": "external",
        "name": "external_heft",
        "command": ["{python}", "-m", "examples.external_heft_scheduler"],
        "working_directory": "..",
        "timeout_seconds": 10,
        "max_output_bytes": 4194304
      }
    ]
  }
}
```

`{python}` 会替换为当前 TriSched 解释器。相对 `working_directory` 以配置文件所在目录为基准；上例用于位于 `configs/` 的配置，因此 `..` 指向仓库根目录。命令始终以参数数组和 `shell=False` 执行，不支持 shell 管道、重定向或字符串拼接。

运行方式与内置策略相同：

```powershell
python -m trisched pipeline --config configs/with_external.json --output outputs/external
```

外部 scheduler 的 makespan、ratio 和 runtime 会以 `<name>_makespan`、`<name>_ratio`、`<name>_runtime_ms` 列加入逐实例 CSV，并在 `summary.json` 使用相同统计口径。

## 3. 进程与传输约定

每个 Scenario 启动一个新进程：

- stdin：一个 UTF-8 JSON 请求，读取到 EOF；
- stdout：只能包含一个 UTF-8 严格 JSON 响应，可带首尾空白；
- stderr：允许诊断文本，成功时不参与结果；
- exit code：0 表示进程层成功，非 0 表示 `scheduler_process_failed`；
- 超时：超过 `timeout_seconds` 后终止进程并报 `scheduler_timeout`；
- stdout 上限：完成后超过 `max_output_bytes` 会被拒绝；
- 生命周期：进程不得依赖前一个 Scenario 的内存状态。

## 4. 请求与响应

请求对象只允许三个字段：

```json
{
  "protocol_version": 1,
  "scheduler_name": "external_heft",
  "scenario": {
    "id": "test-0000",
    "seed": 20260716,
    "tasks": [],
    "resources": [],
    "edges": [],
    "bandwidth": [],
    "latency": []
  }
}
```

实际 `scenario` 是完整的 `Scenario.to_dict()`，上例中的空数组只用于展示字段，不是一份可运行场景。

响应对象必须且只能包含以下五个字段：

```json
{
  "protocol_version": 1,
  "scheduler_name": "external_heft",
  "scenario_id": "test-0000",
  "makespan": 12.5,
  "entries": [
    {"task_id": 0, "resource_id": 1, "start": 0.0, "finish": 2.0}
  ]
}
```

约束：

- `protocol_version` 必须严格为整数 1，不能是布尔值；
- `scheduler_name` 和 `scenario_id` 必须与请求一致；
- task/resource ID 必须是 JSON 整数，时间必须是有限 JSON number；
- response 和每个 entry 都拒绝未知字段；
- 每个任务必须恰好一次，资源和时间合法，依赖、通信和资源互斥成立；
- `makespan` 必须等于最大完成时间；
- 外部结果会依次通过生产 validator 和不复用生产环境时间逻辑的独立 validator。

参考实现位于 `examples/external_heft_scheduler.py`。它只演示协议和进程边界，不作为新的独立算法基线。

## 5. 稳定错误契约

Python adapter API 抛出 `SchedulerAdapterError`，`to_dict()` 固定包含 `code`、`message`，并按上下文增加 `scheduler`、`scenario_id` 和 `details`。统一 evaluator 会逐实例捕获非 HEFT scheduler 错误并写入 CSV/JSONL；CLI 在完整产物写出后以 `evaluation_contains_failures` 汇总，在 stderr 输出单行 JSON 并返回退出码 3。配置构建或 HEFT 基线失败仍立即返回结构化错误。

| code | 含义 | 典型 details |
| --- | --- | --- |
| `scheduler_invalid_spec` | 配置、名称、命令、超时或目录非法 | `missing`、`unknown` |
| `scheduler_unknown` | registry 中不存在该内置 scheduler | `available` |
| `scheduler_duplicate` | registry 或评测列表名称重复 | 无 |
| `scheduler_launch_failed` | 进程无法启动 | `reason` |
| `scheduler_timeout` | 超过单场景 deadline | `timeout_seconds`、`stderr` |
| `scheduler_process_failed` | 进程非零退出 | `exit_code`、`stderr` |
| `scheduler_output_too_large` | stdout 超过上限 | `actual_bytes`、`max_output_bytes` |
| `scheduler_invalid_utf8` | stdout 不是 UTF-8 | 无 |
| `scheduler_invalid_json` | stdout 不是严格 JSON | 行列、输出摘要 |
| `scheduler_invalid_response` | JSON 字段或协议值不合约 | `path` |
| `scheduler_invalid_schedule` | 完整调度未通过 validator | `reason` |
| `scheduler_baseline_failed` | HEFT 失败，ratio 分母不可定义 | `cause` |
| `evaluation_contains_failures` | 评测已完成但含非基线失败 | `failure_count`、`results` |

示例：

```json
{"ok":false,"error":{"code":"scheduler_timeout","message":"external scheduler exceeded 10 seconds","scheduler":"external_heft","scenario_id":"test-0000","details":{"timeout_seconds":10.0,"stderr":""}}}
```

## 6. 安全与可复现要求

- 外部命令被视为团队已审计的可信代码；adapter 不提供容器、用户、网络或文件系统沙箱。
- 禁止把不可信提交、用户输入或网络内容直接拼入 command；协议使用参数数组并关闭 shell。
- scheduler 名称必须匹配 `^[a-z][a-z0-9_-]{0,63}$`，且在一次评测中唯一。
- command、working directory、deadline 和 stdout 上限必须写入配置并进入 `resolved_config.json`。
- 外部程序不得读取 test 标签或 HEFT 结果；它只从 stdin 接收当前 Scenario。
- stderr 只在失败诊断中截取前 2,000 个字符；不得向其中写入密钥或隐私数据。

## 8. 已知限制与后续

- P0-08 后，非 HEFT scheduler 失败会保留逐实例行、应用冻结惩罚并统计真实 failure rate；HEFT 失败仍明确终止，因为 ratio 分母不可定义。详见[P0-08 失败统计与运行清单](./P0-08失败统计与运行清单.md)。
- 每场景启动一次进程，runtime 包含解释器启动和序列化成本；正式报告必须把 adapter 开销与算法计算时间区分说明。
- stdout 上限在进程完成后检查，不能替代操作系统级内存限制。
- 当前协议只返回完整静态 DAG schedule，不支持动态资源、流式事件、断点续跑或在线动作交互。
- 参考 external HEFT 复用本仓库 HEFT，只用于协议 smoke，不得作为独立正确性 oracle 或论文对比基线。
