## 6. 阻塞缺陷：失败恢复破坏现有 manifest

B 在一份已经成功完成、manifest 可验证的 3-seed 输出上，将 seed 32 的训练状态替换为 seed 31 状态，再执行 `--resume`。实际事件顺序为：

1. seed 31 的最终状态验证通过；
2. pipeline 立即从状态重建并写出 seed 31 输出；
3. seed 32 状态验证失败，返回 `ppo_resume_state_mismatch`；
4. 命令退出，最终 run manifest 没有成功重建；
5. B 将故障注入的 seed 32 状态恢复为原始字节，再按原 manifest 重算全部 artifact；
6. 此时仍有 seed 31 的训练曲线发生字节变化，旧 manifest 不能验证恢复后的目录。

可重复的证据为：

```text
artifact=seed_31_training_curve.json
manifest_sha256=da9a43f1c52cdde708ac0f6989fc2a4066eb7e9fa737efe0a94a5135d1ae7dd4
actual_sha256=8a791f3ff1c626c46e3b358ea3375d61f2abf66b7f0c49e54f6b3a6b35200a5b
```

训练曲线的语义字段未改变，差异来自恢复重写后的 JSON key 顺序；但 manifest 的承诺对象是文件字节和 SHA-256，因此这仍是实质性证据破坏。根因是当前 pipeline 缺少“所有 seed 只读预检通过后才允许任何写入”的事务边界。
