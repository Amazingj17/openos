## 1. 目标与边界

P0-08 解决两个会直接影响比赛可信度的问题：scheduler 失败不能让困难实例从统计中消失；每次运行必须能追溯到代码、配置、数据、checkpoint、依赖和实际结果文件。

本任务不改变调度算法、训练方法或数据生成器，也不声称默认惩罚系数具有论文最优性。它把当前工程口径冻结为可配置、可审计的基线；正式论文实验仍须只在 validation 上选择超时阈值和惩罚系数，test 开始后不得调整。

## 2. 失败计分契约

配置字段：

```json
{
  "evaluation": {
    "failure_penalty_ratio": 10.0
  }
}
```

该值默认为 `10.0`，必须是有限 JSON number 且严格大于 1；布尔值、字符串、无穷值和小于等于 1 的数都会在评测开始前被拒绝。

对场景 `i` 和策略 `p`，HEFT makespan 记为 `H_i`：

```text
成功：raw_ratio(i,p) = makespan(i,p) / H_i
      score_ratio(i,p) = raw_ratio(i,p)

失败：raw_ratio(i,p) = 空
      score_ratio(i,p) = failure_penalty_ratio
      score_makespan(i,p) = failure_penalty_ratio × H_i

mean_ratio(p) = sum(score_ratio(i,p)) / N
failure_rate(p) = failure_count(p) / N
valid_schedule_rate(p) = success_count(p) / N
```

因此失败实例始终保留在 `N` 中，且不会因为原始 makespan 缺失而被均值跳过。`success_mean_ratio` 只用于诊断成功子集；当策略全部失败时它为 JSON `null`，不能替代惩罚后的主 `mean_ratio`。

HEFT 是逐实例 ratio 的分母。若 HEFT 自身失败，评测以 `scheduler_baseline_failed` 明确终止，不为该场景虚构分母。任何非 HEFT scheduler 失败则被逐实例记录，其他 scheduler 和后续场景继续执行。

## 3. 逐实例输出

`<split>_per_instance.csv` 为每个 scheduler 增加以下字段：

| 字段后缀 | 成功 | 失败 |
| --- | --- | --- |
| `_status` | `success` | `failure` |
| `_makespan`、`_ratio` | 原始数值 | 空，禁止伪装成真实调度 |
| `_score_makespan`、`_score_ratio` | 等于原始数值 | 惩罚值 |
| `_penalty_applied` | `False` | `True` |
| `_runtime_ms` | 实际尝试耗时 | 失败前实际耗时 |
| `_error_code`、`_error_message` | 空 | 稳定诊断 |
| `_error_details` | 空 | 紧凑 JSON |

`<split>_failures.jsonl` 每行保存一个结构化失败，包含 split、场景 ID/hash、scheduler、runtime、惩罚值和完整 error。没有失败时文件仍存在但长度为 0，便于 manifest 和自动检查使用固定路径。

每个策略的摘要新增：

- `success_count`、`failure_count`、`failure_rate`、真实 `valid_schedule_rate`；
- `success_mean_makespan`、`success_mean_ratio`；
- 使用惩罚分的 mean/std/CI/median/P95 与 win/tie/loss；
- `error_counts`，按稳定错误码计数。

`summary.json` 和 `evaluation_summary.json` 的 `format_version` 升为 2，并明确记录计分定义。为兼容既有报告，`mean_ratio` 和 `mean_makespan` 继续存在，但现在都包含失败惩罚；未经 `failure_rate` 共同解释不得单独引用。

## 4. CLI 退出规则

评测发现非基线失败时，先完整写出 CSV、JSONL、摘要和 manifest，再在 stderr 输出一行：

```json
{"ok":false,"error":{"code":"evaluation_contains_failures","message":"evaluation completed with scheduler failures","details":{"failure_count":20}}}
```

CLI 返回退出码 3，使 CI 和发布流程不会把带失败的 run 当成成功；Python API 的 `run_pipeline`/`evaluate_checkpoint` 仍返回摘要路径，允许上层读取完整失败证据。配置构建失败或 HEFT 基线失败仍立即返回结构化错误。

## 5. `run_manifest.json`

pipeline 和 checkpoint evaluation 都在所有结果写完后生成 manifest。它包含：

| 区域 | 内容 |
| --- | --- |
| `code` | Git commit、工作树是否有未提交修改（含未跟踪文件）、来源是 Git/环境变量/不可用 |
| `runtime` | TriSched、Python、NumPy 和平台版本 |
| `inputs.config` | 原配置名称/hash、`resolved_config.json` hash |
| `inputs.datasets` | 每个 split 的数量和场景 hash 列表聚合 hash |
| `inputs.checkpoint` | checkpoint 名称和 SHA-256 |
| `inputs.dependency_lock` | `requirements-lock.txt` SHA-256；不可用时为 `null` |
| `scoring` | 失败惩罚、失败保留分母、零失败发布要求 |
| `artifacts` | 每个声明输出的相对 POSIX 路径、字节数和 SHA-256 |

manifest 不把自身列入 `artifacts`，避免自哈希循环。摘要只保存固定路径 `run_manifest.json`；验证方读取 manifest 后逐文件重算 hash。路径不记录个人绝对目录，配置源只记录文件名。

在 GitHub Actions 和 openEuler 容器中，workflow 注入 `TRISCHED_GIT_HEAD` 与 `TRISCHED_GIT_DIRTY=false`，保证非 editable 安装仍能记录准确提交和 clean 状态；本地 openEuler 包装器会在注入前拒绝 dirty 工作树。在源码工作区运行时直接读取 Git；有已修改或未跟踪文件会写 `working_tree_dirty=true`，该本地 run 只能作为开发证据，不能作为最终 release。

## 8. 已知限制与复核结论

- 默认惩罚 10.0 是保守工程口径，不是论文理论值；正式 benchmark 必须在 validation 冻结后写入配置。
- `error_counts` 当前区分所有稳定错误码，尚未单列 timeout rate；可由 `scheduler_timeout / count` 精确导出，P1 正式报告再增加直接字段。
- runtime 包含外部进程启动和序列化开销；跨实现效率比较必须单独说明。
- manifest 证明字节和元数据一致，不证明算法结论正确；最终发布仍需 commit、artifact 和报告双签。
- HEFT 基线失败不产生部分汇总，因为所有 ratio 分母均不可定义；其结构化错误必须保留在 CLI/CI 日志。
