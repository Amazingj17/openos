## 2. 策略、seed 与参考口径

主策略固定为 `masked_mlp`，参考策略固定为 `heft`。`score_ratio` 定义为：

```text
policy_makespan / heft_makespan
```

数值越低越好，HEFT 逐行必须合法成功且 `ratio = score_ratio = 1`。HEFT 失败时分母不可定义，报告器拒绝 evidence，不能把 HEFT 失败伪装成惩罚分。

| 策略 | 角色 | 冻结 seed | 解释 |
| --- | --- | --- | --- |
| HEFT | reference baseline | `0` | 确定性参考，逐行必须为 1 |
| Greedy-EFT | baseline | `0` | 确定性启发式 |
| CPOP | baseline | `0` | 确定性启发式 |
| Random | baseline | `20260717..20260721` | 与主策略相同 5-seed 数量 |
| BC | baseline | `20260717` | 已有单 seed 基线，不冒充 5-seed 结论 |
| masked MLP | primary | `20260717..20260721` | 最终主策略要求 5 seeds |
| task-GNN | ablation | `20260717..20260719` | 只使用已完成的 3-seed 消融，不补跑新变量 |

报告器要求每个策略在每个纳入切片上恰好提供 `required_seeds × scenarios` 的笛卡尔积。缺行、重复行、额外 seed、场景 ID/hash 错配都会失败。不同策略 seed 数量不同是契约的一部分，不得复制单 seed 行伪造 5-seed 证据。

### 3.3 `ood_ccr`

## 4. evidence 契约

顶层 evidence 必须包含：

```text
format_version = 1
mode = development | final_test
contract_sha256 = 契约 canonical JSON SHA-256
code.commit = 40 位小写 Git commit
code.working_tree_dirty = boolean
test_accessed = false（development）| true（final_test）
slice_manifests = 每个切片的 scenario_count + scenario_set_sha256
records = 完整逐策略、逐 seed、逐场景记录
records_sha256 = records 的 canonical JSON SHA-256
```

每条 record 必须包含：

```text
slice_id, policy, seed, scenario_id, scenario_hash
status, ratio, score_ratio, penalty_applied
runtime_ms, illegal_action_count, error_code
```

成功行必须满足：

```text
status = success
ratio 为有限正数
score_ratio == ratio（绝对容差 1e-9）
penalty_applied = false
error_code = null 或空字符串
illegal_action_count = 0
```

失败行必须满足：

```text
status = failure
ratio = null
score_ratio = 10.0
penalty_applied = true
error_code 为非空字符串
illegal_action_count 为非负整数
```

失败行保留在主指标分母中。`scheduler_timeout` 计入 timeout count/rate。主结果发布仍要求主策略所有切片零失败、零非法动作。

`runtime_ms` 冻结为“仅 scheduler 调用的 wall-clock 时间”，排除场景加载和报告 I/O；报告 P50/P95。聚合器验证数值和聚合口径，但不替 evidence producer 计时，A 必须在下一阶段复核计时边界。

## 5. 统计方法

每个 policy/slice 报告：

- record/scenario/seed 数量；
- mean、按 seed 稳定排序的全部 `seed_mean_ratios`、seed mean 的总体标准差、median/P50、P95；
- 相对 HEFT=1 的 win/tie/loss；
- failure、illegal action、timeout；
- runtime mean/P50/P95；
- seed→scenario 双层 bootstrap 的 95% CI。

bootstrap 重复 `10,000` 次，随机 seed 固定为 `20260717`。每次先有放回抽取 seed，再有放回抽取共享场景，输出 mean ratio 的 2.5%/97.5% 分位数。

此外，主策略会与每个 comparator 单独比较：

1. 分别对该策略的全部 seed 求每个场景的 seed mean；
2. 计算 `primary - comparator` 的场景配对 delta，负数表示主策略更优；
3. bootstrap 中主策略 seed 与 comparator seed 独立重采样，场景索引保持共享配对；
4. 输出配对 delta、场景级 win/tie/loss、95% CI 和 `supports_primary_improvement`；
5. 只有两个策略 seed 集完全相同时，才额外输出全部 seed×scenario 精确配对统计。

不同 seed 数量的比较估计的是“各策略 seed 分布均值之差”，不是同 seed 训练轨迹的逐项配对。当前不做多重比较校正，因此单个 comparator 的 CI 不应被扩张解释为所有比较同时显著。

## 7. 一次性公开 test 工作流

正式评测前，A/B 共同生成授权 JSON，内容至少绑定：

```text
contract_sha256
release_commit
working_tree_dirty = false
test_slice_id = public_test
authorized_by = [A, B]
authorized_at_utc = YYYY-MM-DDTHH:MM:SSZ
authorization_nonce（64 位小写 hex）
```

契约冻结 `authorization_time_format=utc_seconds_z`。claim 与 final receipt 均严格解析秒级 UTC `Z` 格式；任意文本、缺少 `Z`、`+00:00` offset、无效日期、小数秒和非零填充日期都会被拒绝，合法闰日可通过。

紧邻 public-test loader 之前执行：

```powershell
python -m trisched claim-test-gate `
  --contract configs/p1_b02_evaluation_contract.json `
  --authorization <authorization.json> `
  --receipt <release-artifacts/public_test_gate_receipt.json>
```

claim 先要求传入路径 basename 严格等于契约冻结的 `receipt_name`，再使用 exclusive-create 原子占有 receipt 路径并在返回前 `fsync`；receipt payload 也写入该名称。同一路径已存在 receipt 时第二次或并发 claim 被拒绝。最终报告还必须传入 receipt，并再次核对路径 basename、receipt 内名称和 UTC 时间：

```powershell
python -m trisched build-report `
  --contract configs/p1_b02_evaluation_contract.json `
  --evidence <final-evidence.json> `
  --test-receipt <release-artifacts/public_test_gate_receipt.json> `
  --output <empty-final-report-directory>
```

当前 receipt 只是本地工作流约束，不是密码学不可抵赖机制：签字是冻结姓名列表而非数字签名，删除 receipt 后仍可绕过“同路径一次性”。正式 loader 接入时必须在读取 test 前直接调用 claim，并将授权、receipt 和 release commit 保存到不可覆盖的 CI/release artifact；未完成该接入前不得执行公开 test。

## 16. B 的 OOD producer R1/R2 修复候选

B 以 A 的退回复核记录为唯一输入，只修改失败语义和对应测试，没有重新物化场景。`run_policy()` 现在抛出仍兼容 `ValueError` 的类型化 `IllegalActionError`；`PolicySchedulerRunner` 将其转换为含稳定 task/resource 诊断的 `scheduler_invalid_schedule`。evidence producer 对该 code 固定记录 `illegal_action_count=1`，adapter 与进程内策略进入相同自动报告分项。

producer 在停止 scheduler-only 计时并完成输入未修改检查后，先验证返回对象是否为 `ScheduleResult`。非参考 runner 返回 dict 等错误类型时写入 `scheduler_invalid_response` penalty 行并记录 `actual_type`；HEFT 的同类错误继续以 `ood_reference_failed` 阻断，且不发布部分 evidence。

新增生产路径测试同时覆盖 adapter 非法、真实 `PolicySchedulerRunner` 非法、四切片自动报告计数、非参考错误类型 penalty 和 HEFT fail-fast。结果为 OOD 专项 `13 passed`、报告器合并 `41 passed`、既有接口兼容回归 `26 passed`、全量 `229 passed`；Black check 与 `compileall` 通过。真实策略、5-seed development/OOD、public test、checkpoint 和训练均未触碰。候选等待用户推送后由 A 从新不可变提交二次复核，P1-B02 仍为部分完成。
