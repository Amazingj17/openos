## 1. 目标与非目标

本任务为调度环境建立第二条正确性检查路径，避免生产环境的时间计算或 HEFT 实现发生错误时，由同一段代码“自证正确”。交付内容包括：

- 独立重算执行时间、通信时间和 HEFT upward rank；
- 独立生成插入式 HEFT 调度；
- 独立验证任务全集、执行时长、依赖、资源互斥和 makespan；
- 将 10 个人工黄金案例、生产 HEFT 和独立 oracle 做三方差分；
- 对随机 DAG 的多种策略做属性测试，并用故障注入证明 validator 能抓错。

本任务不证明 HEFT 全局最优，也不替代 P0-B03 的 CP-SAT/MILP 最优解 oracle。它只证明当前静态 DAG 模型下的调度可行性，并交叉检查 HEFT 的确定性实现。

## 2. 独立性边界

独立实现位于 `trisched/oracle.py`。它读取已校验 `Scenario` 的原始字段，但不调用下列生产路径：

| 不复用的生产组件 | 独立路径的做法 |
| --- | --- |
| `HeterogeneousDagEnv` | 重新维护任务表和每个资源的时间区间 |
| `Scenario.execution_time` | 直接计算 `workload / speed` |
| `Scenario.communication_time` | 直接读取 latency、bandwidth 和 edge data 计算 |
| `compute_upward_ranks` | 独立拓扑排序并逆序递推 upward rank |
| `HeftPolicy` | 独立选择 ready task、计算各资源最早完成时间并确定 tie-break |
| `validate_schedule` | 独立重建任务索引、资源区间和依赖约束 |

测试会临时把两个 `Scenario` 时间辅助方法替换成“调用即失败”的函数；独立 HEFT 和独立 validator 仍能完成，以自动守住这条边界。

两条实现仍共享 `Scenario` 数据结构和同一调度数学定义。这是有意保留的接口边界，不代表对输入解析和场景 schema 的独立验证。

## 3. 校验契约

`validate_schedule_independent` 对每份调度依次检查：

1. 场景 ID 一致，entry 数量等于任务数；
2. 每个任务恰好出现一次，任务和资源 ID 均在范围内；
3. 开始/完成时间有限、非负且完成晚于开始；
4. `finish - start` 与任务在所选资源上的执行时间一致；
5. 每条依赖满足父任务完成和跨资源传输后，子任务才能开始；
6. 同一资源上的任意两个执行区间不重叠；
7. 报告 makespan 有限，且等于所有任务完成时间的最大值。

默认绝对容差是 `1e-7`。验证成功后返回任务数、资源数和重算 makespan；失败时抛出 `IndependentValidationError`，并指出首个确定的错误类别。`run_policy` 已在原 validator 之后强制调用该独立 validator，因此所有现有策略和后续学习策略都经过双重检查。

## 4. 三方差分机制

测试对 `tests/fixtures/golden/heft_cases.json` 的 10 个案例逐例执行：

```text
A 的人工推导 fixture
        ↕ ranks / 决策顺序 / 资源 / start / finish / makespan
生产 compute_upward_ranks + HeftPolicy
        ↕ 同一组字段
B 的 independent_upward_ranks + independent_heft_schedule
```

所有浮点字段使用 fixture 声明的 `1e-9` 绝对容差，整数 ID 和决策顺序必须完全一致。当前 10 例全部三方一致，包括插入空隙、非对称链路、固定时延和 tie-break 等高风险边界，因此 P0-A02 可由 B 复核通过。

这项差分能发现生产 HEFT 与人工预期不一致，也能发现人工 fixture 与两套实现同时不一致；若两套实现以相同方式误读数学定义，仍需 P0-B03 的参考 solver 和外部 benchmark 继续降低同源错误风险。

## 5. 属性测试与故障注入

随机属性测试固定生成 40 个可复现 DAG：

- seed：`10000` 至 `10039`；
- 任务数：5 至 12；
- 资源数：1 至 4；
- 额外边概率：0.0 至 0.5；
- 策略：HEFT、Greedy-EFT、Random；
- 验证调度总数：`40 × 3 = 120`。

每份调度都必须通过生产 validator 和独立 validator，并核对报告中的任务数、资源数和 makespan。

故障注入从合法黄金调度出发，每次只注入一个主要错误，覆盖以下 10 类拒绝路径：

| 故障 | 预期检测结果 |
| --- | --- |
| 缺少任务 | entry 数量错误 |
| 重复任务 | 同一任务出现多次 |
| 未知任务 | 任务 ID 越界 |
| 未知资源 | 资源 ID 越界 |
| 非法时间区间 | 完成时间不晚于开始时间 |
| 非有限时间 | NaN 时间戳 |
| 错误执行时长 | 区间长度与 `workload / speed` 不符 |
| 依赖违背 | 子任务过早开始 |
| 资源重叠 | 同一资源的两个任务时间区间相交 |
| makespan 不一致 | 报告值与最大完成时间不同 |

## 8. 已知限制与复核要求

- 当前模型不考虑通信链路竞争、任务抢占、资源故障和动态带宽；validator 与生产环境遵循相同边界。
- 只校验调度可行性和 HEFT 一致性，不校验全局最优性。
- 随机测试是固定 seed 的可复现样本，不是形式化证明。
- 结果对象的结构由内部 dataclass/调用方保证，本 validator 主要诊断调度语义错误。
- 成员 A 应至少复跑全部测试，并独立构造一个依赖违背和一个资源重叠案例；复核完成前 P0-B02 保持“部分完成（提交复核）”。
