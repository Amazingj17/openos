## 4. 正式配置资源预检

### 4.1 参数量

按正式 `hidden_dim=32/message_dim=8` 构造策略，并直接累加参数数组：

| 表示 | 参数量 | 相对 MLP |
| --- | ---: | ---: |
| 14 维 masked MLP | 512 | 1.000× |
| task-GNN v1 | 1008 | 1.96875× |

机器值与公式 `3g² + 6g + (16 + g)h = 1008` 一致。

### 4.2 frozen train state

只加载正式 120 个 train 场景并生成 teacher/frozen state，未加载 test：

| 项目 | 实测值 |
| --- | ---: |
| Scenario / graph 对象 | 120 / 120 |
| frozen transition | 6000 |
| 合法候选总数 | 139,848 |
| 去重 NumPy 张量 | 20,702,976 bytes（约 19.74 MiB） |
| 递归对象估算 | 31,497,972 bytes（约 30.04 MiB） |
| 进程 RSS 结束增量 | 51,789,824 bytes（约 49.39 MiB） |
| 2 ms 轮询 RSS 峰值增量 | 56,012,800 bytes（约 53.42 MiB） |
| `tracemalloc` 峰值 | 33,095,612 bytes（约 31.56 MiB） |

每个 Scenario 的 50 个 step 共用一个 `FrozenTaskGraph`，120 个 Scenario 恰有 120 个 graph 对象，未出现逐 transition 复制邻接矩阵。该内存规模允许启动当前正式配置。

无追踪器的生产式计时为：加载 train `0.496 s`、生成 teacher `17.395 s`、冻结状态 `59.393 s`、合计 `77.284 s`。开启 `tracemalloc` 后冻结增至 `181.804 s`，因此后者只用于内存测量，不作为正常前处理耗时。

### 4.3 CPU 延迟预检

随机初始化的确定性 MLP/task-GNN 在 30 个正式 validation 场景各运行一次，先各做一次 warm-up；计时包含完整 `run_policy` 和 validator：

| 表示 | P50 | P95 | mean |
| --- | ---: | ---: | ---: |
| masked MLP | 73.041 ms | 1656.086 ms | 381.587 ms |
| task-GNN | 81.276 ms | 1741.083 ms | 396.973 ms |

task-GNN/MLP P50 比为 `1.113×`。单轮 Windows 测量方差很大，且使用随机初始化策略；这里只排除数量级失控，不能代替正式 checkpoint 的固定 CPU、重复轮次、配对 P50/P95 报告。
