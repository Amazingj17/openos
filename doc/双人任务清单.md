## 1. 使用规则

- 状态只使用：`已完成`、`部分完成`、`待开始`、`阻塞`。
- 每项任务只有一名主责人，另一人为复核人；复核通过后才能从“部分完成”改为“已完成”。
- 环境转移、HEFT、数据划分、主结果、最终 checkpoint 和提交包不得自审。
- 每人同时只执行一个主任务；P0 未通过时不得开始 P1，P1 未通过时不得开始 P2。
- 任务完成后，主责人按第 7 节提交结果，复核人验收，并在[任务日志](./任务日志.md)追加记录。

## 7. 提交结果规范

### 7.1 一个任务的一次标准提交

每次提交必须包含：

1. **任务标识**：任务 ID、主责人、复核人、目标和明确不做的内容。
2. **变更清单**：新增/修改文件及关键接口，禁止只写“优化代码”。
3. **复现命令**：从仓库根目录可复制执行，不含个人绝对路径。
4. **验证证据**：测试的实际输出；实验任务还要给出基线、样本数、随机种子和前后指标。
5. **产物位置**：代码、配置、checkpoint、结果文件、文档及其 hash。
6. **已知限制**：失败案例、风险、未完成项和回退方法。
7. **复核结论**：`通过`或`退回`，复核人和日期必须明确。

### 7.2 Git 与评审规范

- 首个任务 P0-01 完成前，不接受分散文件作为正式成果。
- 分支名：`feat/<TASK-ID>-short-name`、`fix/<TASK-ID>-short-name`、`exp/<TASK-ID>-short-name` 或 `docs/<TASK-ID>-short-name`。
- commit：`type(scope): [TASK-ID] 简短结果`，例如 `feat(env): [P0-B02] add independent schedule oracle`。
- 一个合并请求只解决一个任务；描述中必须包含第 7.1 节的七项内容。
- 环境、HEFT、数据划分、主结果和发布包必须由非作者批准；禁止主责人自行合并。
- 禁止提交密钥、个人绝对路径、`.idea`、缓存、临时数据和未核验许可证的大文件。

### 7.3 代码任务验收

至少执行：

```powershell
python -m pytest -q
python -m trisched pipeline --config configs/smoke.json
```

并满足：

- 新功能有正常、边界和失败路径测试；核心调度逻辑不得只测“能运行”。
- 调度合法率必须为 100%；出现一个非法调度即退回。
- 公共接口、配置或输出 schema 变化时，同步更新 README 和相关设计文档。
- 格式化、静态检查和测试均通过；不得依赖未记录的本机环境。

### 7.4 实验任务验收

输出目录统一为 `outputs/<run_id>/`，`run_id` 建议使用 `<日期>-<TASK-ID>-<模型>-s<seed>`。至少包含：

```text
resolved_config.json
dataset_manifest.json
manifest.json
checkpoint.*
training_history.json
validation_per_instance.csv
test_per_instance.csv
summary.json
report.md
```

其中 `manifest.json` 最终必须记录代码 commit、依赖版本、操作系统、数据 hash、配置 hash 和 checkpoint hash。`summary.json` 与图表只能由脚本生成，不得手工修改。主结果至少报告样本数、mean/std/95% CI、P50/P95、win/tie/loss、合法率、训练耗时和推理耗时。

### 7.5 文档和最终提交包验收

- Markdown 链接、命令、文件名和参数必须与冻结 release 一致。
- 引用论文、代码、数据和模型时记录 URL/DOI、访问日期和许可证。
- 大型数据与 checkpoint 不直接混入源码提交；提供下载位置、checksum 和离线备份。
- 最终包必须在干净 openEuler 环境完成一次盲审复现，并保留终端日志。
- 两人都能在不修改代码的情况下运行完整 pipeline、解释主结果并完成 5 分钟演示。
