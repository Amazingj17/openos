# TriSched 详细使用说明

本文面向第一次运行 TriSched 的开发者，说明环境安装、数据准备、Masked MLP 与 Task-GNN 的训练和比较、结果文件解读、断点续训、复现实验与常见错误。项目背景、正式实验结论和完整证据索引见[项目 README](../README.md)。

## 1. 项目能做什么

TriSched 是一个静态异构 DAG 调度研究框架。当前代码包含：

- Scenario JSON 校验、确定性生成、内容 hash 和冻结数据划分；
- HEFT、CPOP、Greedy-EFT、Random 及外部 scheduler adapter；
- 独立调度合法性验证器和小图精确 oracle；
- Masked MLP、Task-GNN、HEFT 行为克隆、masked PPO/GAE；
- 多 seed validation、逐实例结果、配对 bootstrap 和失败统计；
- JSON、CSV、SVG、HTML 报告及 SHA-256 manifest；
- epoch 边界断点续训和一键模型比较。

正式学习路径使用两种策略模型：

1. **Masked MLP**：对每个 ready task–resource 候选计算 14 维特征，经 MLP 输出动作分数，再用合法动作 mask 排除不可选动作。
2. **Task-GNN**：保持相同 14 维候选特征、数据、奖励、BC/PPO 和选模口径，增加任务 DAG 消息传递表示；资源关系仍通过候选特征表达。

两种模型都先学习 HEFT teacher，再使用 clipped PPO/GAE 训练。模型比较入口会在训练前检查两边是否使用相同 benchmark、seed、特征、BC、PPO 和 validation selection；任一关键口径不同都会拒绝比较。

## 2. 运行边界

使用前必须明确以下边界：

- 当前环境处理静态 DAG；单个 episode 内资源和链路参数不变化。
- 目标函数是 makespan，主要指标是相对 HEFT 的 `mean_ratio`，数值越低越好，`1.0` 表示与 HEFT 持平。
- Masked MLP 与 Task-GNN 的正式 validation 对照区间跨 0，因此主模型仍保留 Masked MLP；不能把 Task-GNN 的较低点估计表述为稳健领先。
- P1-A05 正式候选已按负结果分支关闭，不得借一键工具启动第二候选、改变 seed 或重新调参。
- public test 未授权且未访问。一键比较只运行 train 和 validation，不会加载 public-test 字节。
- `configs/smoke.json` 使用合成数据，其中名为 test 的 split 只是本地闭环测试，不等于公开 STG public test。

## 3. 环境安装

### 3.1 基本要求

- Python 3.10 或更高版本；
- PowerShell、Bash 或兼容终端；
- NumPy；
- 运行测试时需要开发依赖中的 pytest、Black 等工具。

从仓库根目录安装：

```powershell
python -m pip install -r requirements.txt
```

需要严格复现依赖时使用锁文件：

```powershell
python -m pip install --no-deps -r requirements-lock.txt
python -m pip install --no-build-isolation --no-deps .
```

确认 CLI 可用：

```powershell
python -m trisched --help
```

### 3.2 最小验证

先运行测试和 synthetic smoke：

```powershell
python -m pytest -q
python -m trisched pipeline --config configs/smoke.json
```

smoke 默认写入 `outputs/smoke/`。成功结束时终端会显示 test `mean_ratio` 和 `summary.json` 路径。

## 4. 获取公开 STG 数据

正式 Masked MLP / Task-GNN 配置依赖冻结的 STG rnc50 hetero 数据。原始数据不提交 Git，需要先获取：

```powershell
python scripts/fetch_stg_benchmark.py
```

数据已经下载后可执行离线核验：

```powershell
python scripts/fetch_stg_benchmark.py --offline
```

脚本会验证 archive、逐文件 SHA-256、Scenario projection 和 120/30/30 split。关键文件为：

- [`data/benchmarks/stg-rnc50-hetero-v1.json`](../data/benchmarks/stg-rnc50-hetero-v1.json)：冻结 split 与来源 manifest；
- `outputs/benchmarks/stg-rnc50-hetero-v1/raw/`：本地原始数据目录，不提交 Git；
- [`configs/stg_ppo.json`](../configs/stg_ppo.json)：Masked MLP 三 seed 配置；
- [`configs/stg_task_gnn.json`](../configs/stg_task_gnn.json)：Task-GNN 三 seed 配置。

下载失败或文件 hash 不一致时不要手工修改 manifest，应重新获取正确来源字节。

## 5. 一键训练、验证和比较两种模型

### 5.1 默认命令

数据准备完成后运行：

```powershell
python scripts/run_model_comparison.py --config configs/stg_model_comparison.json
```

该命令按固定顺序完成：

1. 加载并校验一键配置；
2. 校验 Masked MLP 与 Task-GNN 的共享实验契约；
3. 训练 Masked MLP，并为每个 seed 选择 validation best checkpoint；
4. 训练 Task-GNN，并使用相同口径选择 checkpoint；
5. 按相同 seed × 相同 validation scenario 配对评测；
6. 计算均值、分位数、胜平负、分层配对 bootstrap、参数量、训练时间和推理时延；
7. 生成 JSON、三类 CSV、SVG、HTML 和两级 SHA-256 manifest。

默认输出目录由 [`configs/stg_model_comparison.json`](../configs/stg_model_comparison.json) 指定为 `outputs/stg-model-comparison/`。也可显式指定新目录：

```powershell
python scripts/run_model_comparison.py `
  --config configs/stg_model_comparison.json `
  --output outputs/my-model-comparison
```

输出目录非空时，新运行会拒绝混入旧结果。需要重新实验时应换用新的 run 目录；不要覆盖已经冻结的正式输出。

### 5.2 断点续训

异常中断后，使用完全相同的配置和输出目录：

```powershell
python scripts/run_model_comparison.py `
  --config configs/stg_model_comparison.json `
  --resume
```

如果首次运行使用了 `--output`，续训必须传入同一路径：

```powershell
python scripts/run_model_comparison.py `
  --config configs/stg_model_comparison.json `
  --output outputs/my-model-comparison `
  --resume
```

恢复粒度是完整 PPO epoch。入口会比较已保存的 resolved config；输出目录、模型配置、共享契约或数据身份发生变化时会拒绝恢复。当前不支持 minibatch 中间恢复，也不支持把旧状态迁移到不同源码或配置。

### 5.3 一键配置

一键配置包含五个顶层字段：

```json
{
  "format_version": 1,
  "output_dir": "../outputs/stg-model-comparison",
  "masked_mlp_config": "stg_ppo.json",
  "task_gnn_config": "stg_task_gnn.json",
  "comparison": {
    "bootstrap_samples": 10000,
    "bootstrap_seed": 20260717,
    "latency_repeats": 1
  }
}
```

- `output_dir`：顶层结果目录；相对路径按一键配置文件所在目录解析。
- `masked_mlp_config`：Masked MLP 的 BC/PPO 配置。
- `task_gnn_config`：Task-GNN 的 BC/PPO 配置。
- `bootstrap_samples`：分层配对 bootstrap 次数。
- `bootstrap_seed`：bootstrap 的固定随机种子。
- `latency_repeats`：checkpoint 推理时延重复次数；设为 0 可跳过时延测量。

生产对照不应只修改其中一个模型的 seed、特征或训练超参数。若实验目的确实是改变某个变量，应建立新的预注册任务，而不是绕过共享契约检查。

## 6. 模型训练配置说明

### 6.1 公共字段

Masked MLP 与 Task-GNN 配置共同使用：

- `seeds`：训练 seed 列表；
- `benchmark.manifest`：冻结 benchmark manifest；
- `benchmark.raw_root`：本地数据根；
- `features.exclude`：从 16 维候选特征中排除 teacher 泄漏字段，正式输入保持 14 维；
- `behavior_cloning`：hidden dim、epoch、学习率、梯度裁剪和 shuffle offset；
- `ppo`：epoch、每 epoch episode 数、minibatch、actor/value 学习率、GAE、clip、entropy、KL 和梯度裁剪；
- `selection.metric`：固定为 validation mean ratio；
- `selection.failure_penalty_ratio`：失败样本保留在分母中的惩罚值；
- `selection.target_ratio`：相对 HEFT 的目标阈值。

Task-GNN 还包含：

```json
"task_gnn": {
  "architecture": "task_gnn_v1",
  "message_dim": 8
}
```

### 6.2 单独运行 Masked MLP

```powershell
python -m trisched train-ppo --config configs/stg_ppo.json
```

指定输出或恢复：

```powershell
python -m trisched train-ppo `
  --config configs/stg_ppo.json `
  --output outputs/my-mlp-run `
  --resume
```

### 6.3 单独运行 Task-GNN

```powershell
python -m trisched train-task-gnn --config configs/stg_task_gnn.json
```

指定输出或恢复：

```powershell
python -m trisched train-task-gnn `
  --config configs/stg_task_gnn.json `
  --output outputs/my-task-gnn-run `
  --resume
```

### 6.4 只训练行为克隆基线

```powershell
python -m trisched train-bc --config configs/stg_bc.json
```

该入口冻结 HEFT teacher 数据并选择 BC checkpoint，不执行 PPO。

## 7. 一键输出目录

成功运行后目录结构如下：

```text
outputs/stg-model-comparison/
├── resolved_comparison_config.json
├── comparison_pipeline_summary.json
├── comparison_pipeline_manifest.json
├── masked_mlp/
│   ├── ppo_summary.json
│   ├── ppo_run_manifest.json
│   └── ... checkpoints、epoch 状态和 validation artifact
├── task_gnn/
│   ├── task_gnn_summary.json
│   ├── task_gnn_run_manifest.json
│   └── ... checkpoints、epoch 状态和 validation artifact
└── results/
    ├── comparison.json
    ├── comparison_per_instance.csv
    ├── comparison_per_seed.csv
    ├── comparison_per_scenario.csv
    ├── comparison.svg
    ├── comparison.html
    └── comparison_manifest.json
```

关键文件用途：

| 文件 | 用途 |
| --- | --- |
| `resolved_comparison_config.json` | 记录实际配置、数据身份、共享契约与输出路径 |
| `comparison_pipeline_summary.json` | 一键流程入口摘要，指向两模型和所有结果 |
| `comparison.json` | 配对 validation 统计、bootstrap、效率、门禁和 recommendation |
| `comparison_per_instance.csv` | 每个 seed × scenario 的 MLP/GNN ratio、差值和胜平负 |
| `comparison_per_seed.csv` | 逐 seed mean、P50、P95 与胜平负 |
| `comparison_per_scenario.csv` | 每个 scenario 跨 seed 的均值差 |
| `comparison.svg` | 静态对比图，可直接插入文档或演示 |
| `comparison.html` | 汇总指标、图表、结论和结果文件链接 |
| `comparison_manifest.json` | `results/` 内报告文件的 bytes 与 SHA-256 |
| `comparison_pipeline_manifest.json` | 两模型摘要、manifest、报告和图表的顶层 SHA-256 清单 |

`comparison_manifest.json` 负责结果级完整性，顶层 manifest 再绑定训练摘要与结果 manifest。交付前应同时重算两级 hash。

## 8. 指标与图表解读

### 8.1 Ratio

成功调度的单实例 ratio 定义为：

```text
score_ratio = model_makespan / HEFT_makespan
```

- `< 1.0`：该实例上优于 HEFT；
- `= 1.0`：与 HEFT 持平；
- `> 1.0`：劣于 HEFT。

失败样本使用配置中的 penalty ratio，并保留在总样本分母中，不能只报告成功子集。

### 8.2 配对差值

比较固定为：

```text
delta = Task-GNN ratio - Masked MLP ratio
```

因此 delta 越小越有利于 Task-GNN。胜平负也按同一方向统计，不能在报告时反转。

### 8.3 Bootstrap 区间

报告对 training seed 和 scenario 索引做分层配对重采样。只有置信区间完全位于 0 以下，才能支持 Task-GNN 在冻结口径下具有稳健配对优势。区间跨 0 时只能表述为“点估计更低，但差异未被稳健证实”。

当前冻结 validation 结果为：

- Masked MLP mean ratio：`0.723193`；
- Task-GNN mean ratio：`0.690460`；
- Task-GNN−MLP delta：`-0.032733`；
- 95% CI：`[-0.083063, 0.007113]`。

区间跨 0，所以项目继续保留 Masked MLP。HTML/SVG 只负责展示这些冻结口径，不能替代统计门禁。

### 8.4 时延和训练耗时

训练 wall time 用于工程成本比较，不参与 checkpoint selection。推理时延受 CPU、系统负载和重复次数影响，应同时报告运行环境与 `latency_repeats`，不能把单次本地测量解释为跨平台性能结论。

## 9. 查看与核验可视化

直接用浏览器打开：

```text
outputs/stg-model-comparison/results/comparison.html
```

无需浏览器时，可直接查看 `comparison.svg`，或用 XML parser 检查其格式。验收时至少检查：

1. 页面标题、模型名称和 mean ratio 是否与 `comparison.json` 一致；
2. delta 和 95% CI 方向是否为 Task-GNN−MLP；
3. recommendation 是否与区间和门禁一致；
4. 三类 CSV 及 SVG 链接是否使用相对路径；
5. manifest 中的 bytes 和 SHA-256 是否可重算；
6. 页面是否错误宣称 public-test 或整体 OOD 领先。

## 10. 结果完整性核验

PowerShell 可用以下方式重算单个文件：

```powershell
Get-FileHash -Algorithm SHA256 `
  outputs/stg-model-comparison/results/comparison.json
```

批量交付时应逐项核对 `comparison_manifest.json` 与 `comparison_pipeline_manifest.json`。不要手工修改自动生成的 JSON、CSV、SVG 或 HTML；任何内容改变都会导致 hash 漂移，应从同一输入重新生成。

顶层摘要和 manifest 都必须保持：

```json
"test_accessed": false
```

若发现 `test_accessed=true`、public-test 路径、共享配置漂移或缺失 artifact，应拒绝该结果包。

## 11. Scenario 和 checkpoint 常用操作

### 11.1 生成合成场景

```powershell
python -m trisched generate `
  --config configs/smoke.json `
  --output outputs/generated-scenarios
```

### 11.2 校验单个 Scenario

```powershell
python -m trisched validate-scenario `
  --input path/to/scenario.json
```

合法输入退出码为 0；非法输入返回结构化错误，包括错误 code 和 JSON path。

### 11.3 从 checkpoint 复评

synthetic smoke checkpoint 可执行：

```powershell
python -m trisched evaluate `
  --config configs/smoke.json `
  --checkpoint outputs/smoke/masked_mlp.npz `
  --split validation `
  --output outputs/smoke-checkpoint-eval
```

公开 STG 的 public test 受一次性 gate 约束。没有冻结 release commit 和 A/B 授权时，只能运行 validation，不能把 `--split test` 用于公开 STG 数据。

## 12. 常见错误

### 12.1 `refusing to mix comparison runs`

原因：未使用 `--resume`，但输出目录已经包含文件。

处理：为新实验指定新的 `--output`；若确为同一中断运行，使用相同配置和目录加 `--resume`。不要为了绕过检查手工拼接两个 run。

### 12.2 `resume output does not exist or is empty`

原因：对空目录或不存在的目录使用 `--resume`。

处理：去掉 `--resume` 启动新运行，或指向真实的中断目录。

### 12.3 `missing resolved_comparison_config.json`

原因：目录不是由一键入口创建，或关键恢复文件已被删除。

处理：不能安全续训；使用新输出目录重新运行。

### 12.4 `comparison config, model contract, or output path changed`

原因：续训时配置、数据、契约或输出路径与首次运行不同。

处理：恢复原始配置；若确需改变实验，建立新 run，不能复用旧状态。

### 12.5 `must use the same comparison contract`

原因：两模型的 seed、特征、BC、PPO 或 selection 不一致。

处理：对照 [`configs/stg_ppo.json`](../configs/stg_ppo.json) 和 [`configs/stg_task_gnn.json`](../configs/stg_task_gnn.json)，恢复单变量比较口径。

### 12.6 `must use the same benchmark bytes`

原因：两种模型指向不同 manifest 或 raw root。

处理：重新运行 benchmark 获取/离线核验，并让两份配置绑定同一冻结数据根。

### 12.7 找不到 benchmark 文件

先执行：

```powershell
python scripts/fetch_stg_benchmark.py --offline
```

若离线核验仍失败，说明本地数据不完整，需要重新获取；不要创建空文件或修改 manifest hash。

### 12.8 HTML 能打开但图表或链接缺失

确认 `comparison.html`、`comparison.svg` 和 CSV 保持在同一 `results/` 目录，并重算 manifest。移动单个文件会破坏相对链接；交付时应保留完整结果目录。

## 13. 开发和提交前检查

本次模型比较相关的最低检查为：

```powershell
python -m pytest -q `
  tests/test_comparison_reporting.py `
  tests/test_model_comparison_pipeline.py

python -m pytest -q tests/test_documentation.py
python -m pytest -q

python -m black --check `
  scripts/run_model_comparison.py `
  scripts/compare_task_gnn_mlp.py `
  trisched/visualization.py `
  tests/test_model_comparison_pipeline.py

python -m compileall -q trisched scripts tests
python -m json.tool configs/stg_model_comparison.json
git diff --check
```

端到端测试必须实际训练两种微型模型，并核验：

- 两模型 summary 和 run manifest；
- 配对 comparison JSON；
- per-instance、per-seed、per-scenario CSV；
- SVG 可解析；
- HTML 内容和相对链接正确；
- 结果级与流水线级 manifest hash 可重算；
- `test_accessed=false`。

## 14. 复现与审计建议

每次需要保留的最小信息包括：

- Git commit；
- Python、NumPy、操作系统版本；
- benchmark manifest 与 raw 数据 hash；
- 两份模型配置及一键配置 hash；
- seed、BC/PPO 和 selection；
- 两模型 checkpoint、训练摘要和 run manifest；
- comparison JSON、三类 CSV、SVG、HTML；
- 两级 manifest；
- 实际测试输出和是否访问 public test。

主责人自测通过后，复核人仍应从不可变提交和独立输出目录运行微型一键流程、重算 hash，并目视检查 HTML/SVG。完成独立复核前，任务状态只能是“部分完成/提交复核”。当前任务状态见[任务日志](./任务日志.md)和[双人任务清单](./双人任务清单.md)。
